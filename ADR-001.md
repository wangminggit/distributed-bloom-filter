## 决策事项

### 1. Raft 共识实现

**选项**:
- A. HashiCorp Raft 库
- B. etcd/raft 库
- C. 自研 Raft 实现

**决策**: ✅ **A. HashiCorp Raft 库**

**理由**:
- 成熟稳定，生产环境验证
- 文档完善，社区活跃
- 降低开发风险，节省 3-4 周开发时间
- FSM 接口设计清晰，易于集成

**影响**:
- 需要按照 HashiCorp Raft 的 FSM 接口设计状态机
- 依赖 `github.com/hashicorp/raft`

### 2. 分片数量

**选项**:
- A. 3 分片
- B. 6 分片

**决策**: ✅ **B. 6 分片**

**理由**:
- 单分片数据量更小（1.67 亿 → 8350 万）
- 单分片内存更低（120MB → 60MB）
- 扩容更灵活，粒度更细
- QPS 分散更均匀（单节点从 1.7 万 → 8500）

**影响**:
- 需要 6 个 Storage Node Pod
- 重新计算内存和 QPS 分配
- 一致性 Hash 环需要调整

**资源重新计算**:
```
数据量：10 亿
分片数：6
单分片容量：10 亿 / 6 ≈ 1.67 亿 → 修正：10 亿 / 6 ≈ 1.67 亿（不变，之前计算有误）

修正计算：
- 单分片容量：10 亿 / 6 ≈ 1.67 亿
- 单分片计数器：14.4 × 1.67 亿 ≈ 2.4 亿
- 单分片内存：2.4 亿 × 4 bits ≈ 120 MB

等等，之前 3 分片也是 1.67 亿/分片？重新计算：

3 分片：10 亿 / 3 ≈ 3.33 亿/分片
6 分片：10 亿 / 6 ≈ 1.67 亿/分片

6 分片优势：
- 单分片内存：120MB（3 分片是 240MB）
- 总节点数：6 节点（3 分片×3 副本=9 节点 vs 6 分片×2 副本=12 节点）

副本策略调整：
- 3 分片：每分片 3 副本 = 9 节点
- 6 分片：每分片 2 副本 = 12 节点（成本高）

优化方案：
- 6 分片，每分片 2 副本 = 12 节点（强一致）
- 或 6 分片，每分片 1 副本 + 异步复制 = 6 节点（最终一致）

wm 确认：采用 6 分片 × 2 副本 = 12 节点，保证强一致和高可用
```

### 3. 元数据服务

**选项**:
- A. etcd（独立集群）
- B. K8s ConfigMap + gossip
- C. K8s CRD + 状态机

**决策**: ✅ **B. K8s ConfigMap + gossip**

**理由**:
- 简化架构，无需维护独立 etcd 集群
- 利用 K8s 原生能力
- 节点间 gossip 同步状态，去中心化
- 降低运维复杂度

**实现方案**:
- 分片映射表存储在 ConfigMap
- 节点健康状态通过 gossip 协议同步
- Leader 信息通过 Raft 管理（不需要额外存储）

**ConfigMap 结构**:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: dbf-cluster
  namespace: dbf
data:
  shards.json: |
    {
      "0": {"leader": "dbf-storage-0", "followers": ["dbf-storage-1"]},
      "1": {"leader": "dbf-storage-2", "followers": ["dbf-storage-3"]},
      ...
    }
  config.yaml: |
    shards: 6
    replicas_per_shard: 2
    ...
```

### 5. WAL 加密

**选项**:
- A. 需要加密（AES-256）
- B. 不需要加密

**决策**: ✅ **A. 需要加密（AES-256）**

**理由**:
- 数据安全合规要求
- 防止磁盘泄露导致数据暴露
- AES-256 性能开销可接受

**实现方案**:
- 使用 `crypto/aes` + `crypto/cipher` 标准库
- 加密模式：GCM（认证加密）
- Key 管理：K8s Secret 存储主密钥
- 加密粒度：每条 WAL 记录独立加密

**WAL 记录结构**:
```
┌─────────────────────────────────────┐
│ Header (未加密)                      │
│  - Magic (4B)                        │
│  - Version (2B)                      │
│  - Nonce (12B)                       │
├─────────────────────────────────────┤
│ Encrypted Payload (AES-256-GCM)     │
│  - Operation (1B)                    │
│  - Key (变长)                        │
│  - Timestamp (8B)                    │
│  - Checksum (4B)                     │
├─────────────────────────────────────┤
│ Tag (16B, GCM 认证标签)              │
└─────────────────────────────────────┘
```
